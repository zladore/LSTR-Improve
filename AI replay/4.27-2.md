## 1. 当前代码里的 batch 到底是什么？

你这里有三种 batch 概念，容易混在一起。

### 第一种：正常训练 batch

yaml 里写的是：

```yaml
DATA_LOADER:
  BATCH_SIZE: 16
```

所以正常训练阶段，一个训练 batch 是 **16 个窗口样本**。你的训练代码里也确实是：

```python
batch_size = data[0].shape[0]
det_score = model(*[x.to(device) for x in data[:-1]])
```

也就是说训练时是 16 个窗口一起 forward / backward。

---

### 第二种：BatchInference 的 dataloader batch

在 `do_perframe_det_batch_inference` 里，测试时 batch size 被放大了：

```python
batch_size=cfg.DATA_LOADER.BATCH_SIZE * 16
```

你的 `BATCH_SIZE=16`，所以 BatchInference 的 dataloader 一次会取：

```text
16 × 16 = 256 个滑动窗口
```

但是，这只是为了加快普通 batch inference。

---

### 第三种：TTA 实际更新 batch

关键在这里：

```python
for bs, (session, query_indices, num_frames) in enumerate(zip(*data[-3:])):
    ...
    sample_inputs = [x[bs:bs+1].to(device) for x in data[:-4]]
    optimizer = torch.optim.SGD(cls_params, lr=...)
```

虽然外层 dataloader 一次取了 256 个窗口，但 TTA 模式下，代码又把它拆成 `bs:bs+1`，逐个窗口单独更新。

所以老师问“batch 多少”时，你要这样回答：

> 正常训练 batch size 是 16；BatchInference dataloader 一次读入 256 个滑动窗口；但在当前 TTA 代码中，每个窗口会被单独取出并单独更新 classifier，因此 TTA 参数更新的实际 batch size 是 1 个窗口。

这个非常重要。

## 2. 一个窗口里面到底有多少帧？

你的配置是：

```yaml
LONG_MEMORY_SECONDS: 512
LONG_MEMORY_SAMPLE_RATE: 4
WORK_MEMORY_SECONDS: 8
WORK_MEMORY_SAMPLE_RATE: 1
```

THUMOS 的 fps 是 4。
parser 会把 seconds 转成 length：

```text
LONG_MEMORY_LENGTH = 512 × 4 = 2048
WORK_MEMORY_LENGTH = 8 × 4 = 32
```

然后 long memory 每 4 帧采样一次：

```text
LONG_MEMORY_NUM_SAMPLES = 2048 / 4 = 512
WORK_MEMORY_NUM_SAMPLES = 32 / 1 = 32
```

所以每个 sample window 输入给模型的是：

```text
long memory 512 帧特征 + work memory 32 帧特征 = 544 个时间步
```

三模态时，每个窗口大概是：

```text
visual: [544, 2048]
motion: [544, 1024]
text:   [544, 768]
target: [32, 22]
```

其中 visual 来自 `rgb_kinetics_resnet50`，motion 来自 `flow_kinetics_bninception`，text 是 CLIP-L 768 维。模型输出只对应 work memory，所以输出是：

```text
logits: [32, 22]
```

这和 LSTR 的 forward 逻辑一致：模型先切出 long 部分和 work 部分，最后只对 work memory 的 `output` 做 classifier。

## 3. 测试时滑动窗口怎么预测？

`LSTRBatchInferenceDataLayer` 不是每 32 帧切一段，而是**逐帧滑动**：

```python
for work_start, work_end in zip(
    range(0, target.shape[0] + 1),
    range(self.work_memory_length, target.shape[0] + 1)
):
```

所以对于一个视频，它会构造：

```text
[0, 32]
[1, 33]
[2, 34]
...
```

每个窗口都是 32 帧 work memory。

但是写入预测结果时：

```python
if query_indices[0] == 0:
    pred_scores[session][query_indices] = sample_probs
else:
    pred_scores[session][query_indices[-1]] = sample_probs[-1]
```

意思是：

* 第一个窗口 `[0, ..., 31]`：保存 32 帧预测；
* 后面的每个滑动窗口：只保存最后一帧预测。

所以这是标准 online 逻辑：当前时刻只输出当前帧，不能重复覆盖历史帧。

## 4. 当前 TTA 的 step=2 到底是什么意思？

你现在的 TTA 逻辑是：

```python
for _ in range(TTA_STEPS):
    logits = model(*sample_inputs)[0]
    q = text_calibrator.get_text_target_distribution(...)
    loss = KL(logits, q)
    optimizer.zero_grad()
    loss.backward()
    optimizer.step()
```

所以 `step=2` 的意思是：

> 对当前这个单独的滑动窗口，用 KL loss 连续更新 classifier 两次。

不是训练两个 epoch，也不是对整个测试集更新两轮。
而且每个窗口开始前都会执行：

```python
for p, p0 in zip(cls_params, base_cls_params):
    p.data.copy_(p0.data)
```

也就是说：

> 每个窗口都会把 classifier 重置回原始 checkpoint 参数，再做当前窗口自己的 TTA 更新。

所以当前方法不是“越测越适应”，而是“每个窗口独立做一次临时校准”。

这个设计其实比较安全，因为不会把前面窗口的错误伪标签累积传到后面窗口；但缺点是每次更新只依赖一个窗口，甚至大多数时候只依赖最后一帧，学习率太大时容易过调。

---

## 5. 当前 KL 的目标 Q 是哪里来的？

在 `text_calibration.py` 里，Q 的构造是：

1. 读取当前视频对应的 frame text feature；
2. 根据 `query_indices` 对齐到当前窗口的 32 帧；
3. 用 frame text feature 和 class text feature 做相似度；
4. 除以 `TEXT_TEMP=0.07`；
5. softmax 得到文本分布 Q；
6. 如果是 `tta_kl_topk`，只保留视觉 logits 当前 top-k 类别上的文本分布，其余类置为很小。

所以你可以这样解释：

> Q 不是 ground truth，也不是模型预测本身，而是由当前帧文本特征与类别文本特征相似度得到的文本伪目标；在 `tta_kl_topk` 中，文本目标只在视觉模型当前 top-k 类别内部生效，从而避免文本直接强行拉向所有类别。

当前 `q = q.detach()`，所以 Q 在梯度里是常数目标，梯度只更新 classifier。

## 6. 为什么老师说 step=2 可能就收敛，是合理的

因为你的 TTA 更新范围非常小：

```python
只更新 classifier: Linear(1024 -> 22)
```

模型最后的分类器就是：

```python
self.classifier = nn.Linear(self.d_model, self.num_classes)
```

而你的 `d_model=1024`，`num_classes=22`，所以更新参数量大约只有：

```text
1024 × 22 + 22 = 22550
```

相对整个 LSTR 很小。

在 batch inference 中，每个视频只有第一个滑动窗口使用整个 32 帧 work memory 计算 KL；之后的所有滑动窗口虽然仍然是完整的 32 帧窗口，但由于前 31 帧已经在之前窗口预测过，因此 TTA 和最终保存都只针对最后一帧进行。由于测试阶段采用逐帧滑动窗口，除每个视频第一个窗口外，其余窗口与前一窗口高度重叠，因此当前代码只对每个后续窗口的最后一帧计算 KL loss 并保存预测结果。
：

```python
logits_sel = logits[-1:]
q_sel = q[-1:]
```

也就是说，大部分 TTA 实际是：

```text
用 1 帧的文本伪目标，对 2.25 万个 classifier 参数做 1~2 步 SGD
```

这种情况下，`5e-4` 确实可能偏大，`step=2` 之后已经能把 classifier 往 Q 的方向推很多，再继续加 step 很可能不是“更充分学习”，而是“过度贴合当前窗口伪目标”。

## 8. 一个很重要的小问题：你的 pkl 会被覆盖

当前 inference 结尾保存 pkl 的路径是：

```python
osp.splitext(cfg.MODEL.CHECKPOINT)[0] + '.pkl'
```

这意味着只要你用的是同一个 checkpoint，不同 TTA 设置都会把同一个 `.pkl` 覆盖掉。

所以你跑实验时一定要保存 log，或者跑完马上改名 pkl，例如：

```bash
mv checkpoints/xxx/epoch-25.pkl checkpoints/xxx/epoch-25_tta_topk_step2_lr1e-4_tri.pkl
```

否则后面你想回看每组预测结果，会发现只剩最后一次。
