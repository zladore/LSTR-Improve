下面这版你可以直接记录到实验计划里。核心目标是：**把“训练时辅助损失有用”和“测试时更新有用”分开证明，同时重点验证 online 保留参数在 OAD 视频流里是否有效。** 这个设计逻辑来自 TTT 论文第 3 章的实验对照思路。

# LSTR/OAD 推理时调优实验尝试整理

## 一、主线实验：证明 TTA / TTT 是否真的有效

| 实验编号 | 设置                           | 目的                  | 预期                      |
| ---- | ---------------------------- | ------------------- | ----------------------- |
| A0   | 原始 LSTR baseline             | 作为固定对照              | 例如当前 71.950 / 71.911 锚点 |
| A1   | 训练时加辅助损失，但测试时不更新             | 判断提升是不是只来自训练阶段      | 如果提升，说明辅助任务本身有帮助        |
| A2   | 测试时更新，但每个 window 后 reset     | 验证单窗口 TTT 是否有效      | 稳定，但提升可能有限              |
| A3   | 同一个视频内 online 保留参数，视频间 reset | 最核心实验，模拟真实 OAD 流式测试 | 最可能有效                   |
| A4   | 整个测试集一直 online 保留参数          | 验证跨视频持续适应是否可行       | 风险较高，可能漂移               |

最重要的是 **A0 / A1 / A2 / A3**。
如果结果是：

```text
A0 < A1 < A2 < A3
```

那故事就很完整：
**辅助任务有用，测试时更新进一步有用，视频内 online 保留参数最适合 OAD。**

---

## 二、online 保留范围实验

这个是你刚才最关心的问题：**预测下一个 window 时，分类器/适配参数要不要保留上一个 window 的更新结果。**

| 实验编号 | 保留策略                  | 解释             | 风险         |
| ---- | --------------------- | -------------- | ---------- |
| B1   | 每个 window 更新后恢复原始参数   | 每个窗口独立适应       | 提升小，但安全    |
| B2   | 同一个视频内保留，视频结束 reset   | 最推荐，符合 OAD 视频流 | 动作切换处可能有滞后 |
| B3   | 整个 test set 都保留       | 最激进            | 不同视频之间容易污染 |
| B4   | 每 N 个 window reset 一次 | 折中方案           | 需要调 N      |

优先级：

```text
B2 > B1 > B4 > B3
```

我建议你先做：

```text
baseline
reset-window
online-within-video
online-across-dataset
```

这样就能非常清楚地回答：**online 保留到底有没有帮助，以及保留到什么范围最合适。**

## 三、更新哪些参数的实验

不要一开始更新整个 LSTR。TTT 的思想是更新共享表示，但在你的 LSTR 里，如果更新太多，容易漂移。建议从轻量参数开始。

| 实验编号 | 更新对象                                   | 推荐程度 | 原因                |
| ---- | -------------------------------------- | ---: | ----------------- |
| C1   | 只更新 classifier / text calibration 相关参数 |    高 | 最接近你当前思路，成本低      |
| C2   | 只更新新加的 adapter / prompt / scale 参数     |    高 | 安全，不破坏原模型         |
| C3   | 更新 fusion 层                            |    中 | 可能改善三模态融合，但有漂移风险  |
| C4   | 更新最后一层 temporal block                  |   中低 | 表达能力更强，但风险变大      |
| C5   | 更新整个 LSTR                              |    低 | 容易过拟合当前窗口，不建议主线先做 |

推荐主线从：

```text
只更新轻量 adapter / classifier calibration 参数
```

开始。
不要一开始就更新 backbone 或完整 transformer。

## 四、测试时损失 (l_s) 的设计实验

这是最关键的创新点。你需要证明：**这个测试时损失和动作检测主任务有关，而不是随便构造的自监督损失。**

| 实验编号 | 测试时损失                    | 思路                                         | 风险                     |
| ---- | ------------------------ | ------------------------------------------ | ---------------------- |
| D1   | 文本一致性损失                  | 当前视觉预测分布和动作文本原型分布保持一致                      | 文本原型不准会误导              |
| D2   | RGB / Flow / Text 三模态一致性 | 不同模态对同一 window 的预测应接近                      | 动作边界处可能不一致             |
| D3   | 当前窗口与轻微扰动窗口一致性           | 对输入做 dropout / noise / feature mask，预测保持稳定 | 太强会压制真实变化              |
| D4   | 低熵约束                     | 让预测更确定                                     | 容易强化错误高置信              |
| D5   | 历史稳定预测一致性                | 当前 window 和前面稳定 window 保持一定一致              | 动作切换时可能拖慢反应            |
| D6   | 文本 top-k 分布软目标           | 不强制单一类别，而是用 top-k 文本相似类别做软监督               | 比 hard pseudo label 安全 |

优先推荐：

```text
D1：文本一致性
D2：三模态一致性
D6：文本 top-k 软目标
```

不建议一开始做纯低熵，因为它容易把错误预测变得更自信。

## 五、学习率和更新步数实验

TTT 类方法非常依赖学习率和步数。OAD 里每个 window 都会更新，所以不能太激进。

| 实验编号 |   lr | step | 说明         |
| ---- | ---: | ---: | ---------- |
| E1   | 1e-5 |    1 | 最安全        |
| E2   | 5e-5 |    1 | 推荐起点       |
| E3   | 1e-4 |    1 | 可能有效       |
| E4   | 5e-4 |    1 | 风险较高       |
| E5   | 5e-5 |    2 | 检查多步更新是否更好 |
| E6   | 1e-4 |    2 | 中等风险       |
| E7   | 5e-5 |    3 | 可能开始漂移     |

我建议主线先跑：

```text
lr = 5e-5, 1e-4
step = 1
```

如果有效，再扩展到 step=2。
不要一开始就做很多步更新。

## 六、防漂移策略实验

online 保留参数虽然可能更好，但最大风险是错误累积。所以需要做一些保护机制。

| 实验编号 | 策略                           | 作用               |
| ---- | ---------------------------- | ---------------- |
| F1   | 只在低熵/高置信 window 更新           | 避免不确定样本污染参数      |
| F2   | 只在文本一致性高时更新                  | 视觉和文本不一致时不更新     |
| F3   | 限制每次参数变化范数                   | 防止单个 window 更新过猛 |
| F4   | EMA 参数更新                     | 让 online 更新更平滑   |
| F5   | 动作切换疑似区域不更新                  | 避免边界附近错误平滑       |
| F6   | 每个视频开头 warm-up 几个 window 不更新 | 等模型积累稳定证据        |

优先推荐：

```text
F1：高置信才更新
F2：文本一致性高才更新
F4：EMA 更新
```

## 七、top-k 相关实验

你现在本身也在做 top-k。TTA 和 top-k 可以结合，但要分阶段，不要混在一起乱跑。

| 实验编号 | 设置                             | 目的                 |
| ---- | ------------------------------ | ------------------ |
| G1   | 固定当前最佳 top-k，不做 sweep          | 先验证 TTA 是否有效       |
| G2   | TTA 有效后，再 sweep topk=2/3/4/5   | 看 TTA 是否改变最佳 top-k |
| G3   | online TTA + top-k soft target | 用 top-k 文本类别作为软目标  |
| G4   | reset TTA + top-k soft target  | 对比 online 是否必要     |

优先顺序：

```text
先固定 top-k
再做 top-k sweep
```

否则变量太多，结果不好解释。

## 八、必须记录的结果分析

不要只记最终 mAP。每个实验至少记录这些：

```text
1. 最终 mAP，按百分数记录，例如 71.950
2. 每类 AP
3. 运行命令
4. checkpoint 路径
5. 是否 online 保留参数
6. 是否视频间 reset
7. 更新参数范围
8. 测试时 lr / step
9. 测试时损失类型和权重
10. top-k 设置
11. 输出 pkl 副本路径
12. log 文件名
```

建议 log 命名：

```text
tta_A3_online_video_epoch14_lr5e-5_step1_topk3_textloss.log
tta_A2_reset_window_epoch14_lr5e-5_step1_topk3_textloss.log
tta_A1_joint_no_update_epoch14_topk3_textloss.log
```

## 九、我建议你最先做的 6 个实验

优先不要铺太大，先做最能说明问题的 6 个：

| 顺序 | 实验                 | 设置                        |
| -: | ------------------ | ------------------------- |
|  1 | baseline           | 原始最佳 checkpoint，固定 top-k  |
|  2 | joint-no-update    | 训练时有辅助损失，测试时不更新           |
|  3 | reset-window       | 每个 window 更新 1 step，预测后恢复 |
|  4 | online-video       | 同视频内保留参数，视频间 reset        |
|  5 | online-dataset     | 整个测试集持续保留参数               |
|  6 | online-video + 防漂移 | 高置信/文本一致性满足时才更新           |

如果这 6 个里 **online-video 明显优于 reset-window**，那就说明你这个方向很值得继续做。

## 十、最终可以形成的论文/汇报逻辑

你后面可以这样讲：

```text
我们参考 Test-Time Training 的思想，不直接使用测试标签，
而是在测试阶段利用无标签 window 构造辅助一致性目标，
对轻量分类/适配参数进行在线更新。

为了区分训练时辅助监督和测试时调优的贡献，
我们设置了 baseline、joint-no-update、reset-window 和 online-video 四组对照。

进一步地，我们比较了不同参数保留范围，
验证在 OAD 场景中，同一视频内部保留上一个 window 的更新状态，
是否比每个 window 独立更新更符合流式视频分布。
```

最核心实验结论要围绕一句话展开：

**OAD 是天然流式任务，因此测试时适应不应该只看单个 window，更应该验证同一视频内连续保留参数是否能利用稳定的测试分布信息。**
